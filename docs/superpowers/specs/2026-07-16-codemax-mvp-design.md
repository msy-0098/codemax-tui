# CodeMax TUI MVP 设计规格

- 状态：已完成交互确认，待用户审阅书面规格
- 日期：2026-07-16
- 上游基线：`anomalyco/opencode` 的 `dev` 分支
- 许可证：MIT，发布产物保留上游版权和许可证声明

## 1. 产品目标

CodeMax 是基于 OpenCode 的中文开发者优化版 TUI 编程助手。MVP 首发 Windows x64，同时保持 Bun、TypeScript 和 OpenTUI 架构的跨平台能力，后续可增加 macOS 与 Linux 产物。

MVP 必须实现：

- 完整保留 OpenCode 的 Agent、工具、插件、MCP 和模型供应商能力。
- 默认跟随系统语言并支持手动切换中文、英文。
- 中文模式保留命令、模型名、工具名、API 和文件路径等英文技术术语。
- 中文模式下优先展示 DeepSeek、通义千问、智谱、Kimi、豆包等国内服务商，同时保留完整供应商列表和 OpenAI 兼容接口。
- 提供响应式双栏 TUI、统一中文错误体验、OpenCode 配置迁移、基础测试和私有发布流水线。
- 同时提供 Windows 便携版和安装版；安装版可从快捷方式打开 Windows Terminal 并启动 CodeMax。

## 2. 非目标

以下内容不纳入 MVP：

- 重写 Agent、工具执行、模型协议或会话存储核心。
- 桌面 GUI 或自带终端模拟器。
- macOS、Linux 的正式发行产物。
- 静默自动更新。私有测试期由用户从 Release 手动更新。
- Windows 代码签名证书采购。没有证书时允许出现 SmartScreen 提示，并提供 SHA-256 校验。
- 云端账户、计费、遥测平台或 CodeMax 自有模型网关。

## 3. 上游同步与模块边界

仓库保留 `upstream` 远程指向 OpenCode。CodeMax 主线从上游 `dev` 派生，通过固定周期的 fetch、merge、差异测试吸收更新。

系统分为三层：

1. **OpenCode 上游核心**：Agent、会话、权限、工具、MCP、插件、供应商、存储和 OpenTUI 基础组件。除品牌不可避免的入口改动外，尽量不修改核心行为。
2. **CodeMax 产品定制层**：国际化资源、国内供应商排序、响应式双栏工作台、品牌、配置迁移、错误分类与中文操作建议。
3. **CodeMax 发布层**：Windows 图标、编译参数、便携压缩包、Inno Setup 安装器、校验和、Release 和 Gitee 镜像。

CodeMax 定制优先通过独立资源、配置和小范围组件扩展实现。每次合并上游后，必须运行品牌残留、翻译键、配置兼容和 Windows 产物测试。

## 4. 品牌与图标

品牌名统一为 `CodeMax`，命令为 `codemax`，Windows 可执行文件为 `codemax.exe`。用户提供的 `D:/codemax/ico/CodeMax.png` 是品牌源图；已验证源文件为 1254×1254、24 位 RGB PNG。图标处理只把与画布边缘连通的纯黑外底转换为透明区域，不改动圆角底板、Logo、文字或配色。处理后的主图用于生成：

- 包含 16、24、32、48、64、128、256 像素图层的 Windows `.ico` 文件，并人工检查小尺寸辨识度。
- 安装程序、卸载入口、开始菜单和桌面快捷方式图标。
- README、GitHub/Gitee 私有 Release 的品牌图片。

TUI 不直接渲染 PNG。终端内使用轻量文字 Logo，以及来自源图的深色背景、青色高光和电蓝强调色。终端不支持真彩色或复杂字符时自动降级。

发布包必须包含 OpenCode 的 MIT `LICENSE`，并在 `THIRD_PARTY_NOTICES.md` 中说明 CodeMax 基于 OpenCode 构建且不隶属于 OpenCode 团队。

## 5. TUI 体验

主界面采用已确认的“工作台双栏”方案：

- 左栏展示当前项目文件树、Git 变更和会话导航。
- 右栏展示对话、工具执行、diff 摘要、错误信息和输入框。
- 顶栏紧凑展示 CodeMax、当前 Agent、模型和项目。
- 底栏展示语言、上下文占用、Git 状态和可用操作。
- 文件树、变更列表和技术标识保持稳定宽度，动态文案截断或换行，不推动整体布局跳动。

响应式规则：

- 终端宽度不低于 120 列时启用双栏。
- 80 至 119 列时折叠左栏，通过键盘命令或弹层访问文件和变更。
- 低于 80 列时进入紧凑模式，隐藏非必要状态，仅保留会话、输入和核心操作。
- Windows Terminal 提供完整色彩和字符效果；CMD 使用兼容色板与 ASCII 降级字符。

键盘操作延续 OpenCode 现有习惯；只有在不存在上游快捷键或存在明确冲突时才新增 CodeMax 快捷键。

## 6. 国际化与术语

新增类型安全的消息目录，不在组件内散落中英文条件判断。每个 UI 文案使用稳定 key，英文是完整回退语言，简体中文是首发翻译语言。

语言解析优先级为：

1. 命令行显式参数。
2. CodeMax 用户设置。
3. `LANG`、`LC_ALL` 或 Windows 系统区域。
4. 英文回退。

日期、时间、数量和错误详情使用相同 locale 上下文。命令、参数、模型 ID、供应商 ID、Agent 名、工具名、API、MCP、Git、文件名和路径不翻译。中文文案应说明操作结果和下一步，不直译上游俚语。

CI 检查中英资源 key 完全一致，缺失翻译不能进入 Release。

## 7. 供应商体验

供应商注册、认证和模型调用继续使用上游实现。CodeMax 只增加展示层元数据与中文设置引导：

- 中文模式优先显示 DeepSeek、通义千问、智谱、Kimi、豆包和 OpenAI 兼容接口。
- 英文模式保持上游默认排序，用户最近使用的供应商始终优先。
- 每个国内服务商提供 API Key、Base URL、代理和模型选择的中文提示。
- 不在仓库、配置模板或诊断包中包含真实密钥。
- 供应商功能能力由上游模型元数据决定，不在 CodeMax 中维护第二套模型协议。

## 8. 配置、数据与迁移

CodeMax 使用独立命名空间：

- 全局配置文件为 `%APPDATA%\CodeMax\codemax.jsonc`，同时接受同目录的 `codemax.json`。
- 项目配置文件为 `<项目>\.codemax\codemax.jsonc`，同时接受 `.json` 后缀。
- 自定义配置入口使用 `CODEMAX_CONFIG`、`CODEMAX_CONFIG_DIR` 和 `CODEMAX_CONFIG_CONTENT`。
- 会话、日志、认证和偏好写入 `%LOCALAPPDATA%\CodeMax\` 下的独立子目录。

配置结构尽量兼容 OpenCode schema，但 CodeMax 不直接读写 OpenCode 数据目录。

首次启动时若检测到 OpenCode：

1. 显示检测到的配置、认证和会话类别。
2. 用户明确选择要迁移的类别后才继续。
3. 在临时目录复制并校验，敏感字段不打印到界面或日志。
4. 校验通过后原子写入 CodeMax 目录。
5. 任一步失败都回滚 CodeMax 新写入内容，OpenCode 原数据始终保持不变。

默认迁移配置和认证；会话迁移单独选择；日志不迁移。不创建软链接或共享可写文件。

## 9. 运行数据流

一次正常启动依次经过：

1. 检测终端宽度、颜色能力、系统语言和当前项目。
2. 按命令行参数、环境变量、项目配置、全局配置、默认值加载配置。
3. 选择语言资源并计算供应商展示顺序。
4. 启动上游 Agent 核心，TUI 订阅会话、模型和工具的流式事件。
5. 将会话、认证和用户偏好写入 CodeMax 独立目录。

TUI 只是事件的展示和交互层，不复制 Agent 状态机或工具执行逻辑。

## 10. 异常处理

所有面向用户的错误使用稳定 CodeMax 错误码、中文摘要、建议动作和可展开技术详情。错误详情必须脱敏。

- `CM-NET-*`：DNS、连接、代理、TLS 和超时；提供重试及代理设置入口。
- `CM-AUTH-*`：API Key 无效、登录过期或权限不足；指明供应商并引导重新认证。
- `CM-RATE-*`：限流、余额不足和服务过载；显示可用等待时间并允许切换模型。
- `CM-MODEL-*`：模型缺少工具调用、图片或上下文能力；推荐兼容模型。
- `CM-CONFIG-*`：配置解析、schema 和迁移失败；显示文件、行列并执行迁移回滚。
- `CM-TOOL-*`：工具失败；显示退出码和精简 stderr，可展开完整详情。
- `CM-TERM-*`：终端能力不足；自动切换兼容色板、字符和布局。
- `CM-CRASH-*`：非预期错误；保存可恢复会话并生成脱敏诊断包。

日志不得记录完整 API Key、Authorization header、Cookie、用户提示中的明显凭据或环境变量密钥值。

## 11. Windows 打包与启动

复用上游 Bun 编译流程生成独立的 Windows x64 可执行文件，并保留 baseline CPU 构建。编译阶段注入版本、CodeMax user agent 和 Windows 图标。

便携版包含 `codemax.exe`、许可证、第三方声明和最小使用说明。安装版使用 Inno Setup，提供：

- 安装与卸载入口。
- 可选写入用户级 PATH。
- 开始菜单和可选桌面快捷方式。
- 快捷方式优先调用 Windows Terminal；未安装 Windows Terminal 时回退到系统控制台。
- 卸载默认保留用户配置和会话，另提供明确的“删除用户数据”选项。

MVP 不静默自动升级。启动时可显示非阻断式版本提示，但私有仓库鉴权不能要求用户把仓库令牌写进 CodeMax 配置；私有测试阶段允许关闭版本检查。

## 12. 测试策略

测试从受影响的包目录运行，不从 monorepo 根目录运行测试。

### 静态质量门禁

- 格式、lint 和各受影响包的 `bun typecheck`。
- 中英资源 key 一致性、品牌残留和许可证检查。
- 密钥、令牌和敏感日志模式扫描。

### 单元测试

- 语言解析、英文回退和不可翻译术语。
- 中文与英文供应商排序。
- CodeMax 配置优先级和路径隔离。
- 错误分类、错误码、建议动作和脱敏。
- 终端宽度断点与双栏折叠状态。

### 集成测试

- 使用临时目录验证 OpenCode 配置迁移、选择性迁移、失败回滚和重复运行。
- 使用本地测试服务验证流式模型响应、401、429、超时、断线和取消。
- 验证会话保存、异常退出和恢复。
- 验证 Windows Terminal 启动参数与系统控制台回退。

### TUI 与产物测试

- 对 80、120、160 列的关键界面做稳定快照或像素无关结构断言。
- 在 Windows Terminal 和 CMD 中人工验收中文宽字符、键盘操作、颜色降级和无重叠。
- 在无 Bun、无 Node 的全新 Windows x64 环境运行 `--version`、`--help` 和首次启动冒烟测试。
- 验证安装、PATH、快捷方式、覆盖升级和卸载；卸载不应默认删除用户数据。

## 13. 私有 CI 与发布

GitHub 私有仓库是主仓库，Gitee 私有仓库是镜像。所有凭据仅保存在 CI secret 或本机凭据存储中。已经出现在聊天中的旧 Gitee 令牌不得使用。

Release 流程：

1. 版本标签触发 GitHub Actions。
2. 顺序执行静态检查、单元测试、集成测试、Windows TUI 测试和产物冒烟。
3. 生成 `CodeMax-x64.zip`、`CodeMax-x64-baseline.zip`、`CodeMax-Setup-x64.exe`、`SHA256SUMS.txt` 和 `THIRD_PARTY_NOTICES.md`。
4. 上传 GitHub Private Release，并由维护者完成人工下载验收。
5. 验收通过后同步源代码、标签和 Release 产物到 Gitee 私有镜像。
6. 生成中文变更日志，记录上游基线 commit。

任何门禁失败都不得发布或同步 Gitee Release。

## 14. MVP 验收标准

满足以下条件才视为可交付：

- Windows x64 便携版和安装版能在无开发环境的新系统启动。
- `codemax` 命令、开始菜单快捷方式和卸载入口工作正常。
- 中文/英文可切换并持久化，所有首发界面无缺失资源 key。
- 中文模式国内供应商置顶，其他上游供应商仍可用。
- 120 列及以上显示双栏，窄终端可完整操作且无不可达功能。
- OpenCode 配置只在用户确认后复制；失败回滚且不修改原数据。
- 已定义的网络、认证、限流、配置、工具和终端错误提供中文动作建议。
- 日志和诊断包通过敏感信息脱敏测试。
- 所有测试门禁通过，产物 SHA-256 校验一致。
- GitHub 和 Gitee 私有仓库及 Release 镜像流程完成一次演练。
- Release 包保留 MIT 许可证与上游归属说明。

## 15. 预计排期

以一名熟悉 TypeScript/Bun 的开发者全职投入估算：

- 第 1 天：建立上游同步基线、品牌边界、命令和 Windows 构建。
- 第 2 天：国际化基础设施、语言切换和首批中英文资源。
- 第 3 天：响应式双栏工作台和终端降级。
- 第 4 天：供应商排序、国内服务商中文配置体验和剩余核心翻译。
- 第 5 天：独立配置目录、OpenCode 迁移和统一错误系统。
- 第 6 天：单元/集成/TUI 测试、便携版和 Inno Setup 安装器。
- 第 7 天：GitHub CI、Gitee 私有镜像、发布演练和中文交付文档。
- 第 8 至 9 天：用于上游变化、Windows 兼容问题和验收修复的缓冲。

最快可用版本目标为 7 个工作日，承诺窗口为 6 至 9 个工作日。若需要采购并接入 Windows 代码签名证书，排期单独计算。

## 16. 主要风险与控制

- **上游变化快**：记录基线 commit，定期小批量合并，使用差异测试锁定 CodeMax 行为。
- **全量翻译范围膨胀**：MVP 只覆盖 CLI/TUI 核心路径、配置和常见错误；开发者日志保留英文详情。
- **中文宽字符布局异常**：布局按显示宽度计算，在三档终端宽度和两类 Windows 终端中验证。
- **供应商接口变化**：复用上游 provider 和模型元数据，只维护展示顺序与中文引导。
- **私有镜像凭据泄露**：使用短权限令牌、CI secret、日志脱敏和发布前敏感信息扫描。
- **未签名安装器提示**：私有测试期提供 SHA-256；正式公开发布前接入可信代码签名。
